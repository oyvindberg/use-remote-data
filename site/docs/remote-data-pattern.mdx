# The RemoteData Type

Every async request is in exactly one state: it hasn't started, it's in flight, it failed, or it succeeded.
Most code models this with a bag of booleans and optional values — `isLoading`, `data?`, `error?` — and nothing stops you from having `isLoading: true` and `data: someValue` at the same time.

`RemoteData` is a union type. One state at a time. TypeScript knows which one.

```ts
type RemoteData<T> =
    | { type: 'initial' }                // haven't started yet
    | { type: 'pending' }                // request in flight
    | { type: 'failed'; errors; retry }  // failed — with errors and a retry function
    | { type: 'success'; value: T };      // succeeded — here's your data
```

When you're in the `'success'` branch, `value` is `T`. Not `T | undefined`. Not `T | null`. Just `T`.
No narrowing gymnastics. TypeScript does it for you.

## Stale data is still data

Most libraries throw your data away when they re-fetch. You had a dashboard full of numbers — now you have a spinner.

`use-remote-data` keeps the old value visible while the refresh happens. Three additional states handle this:

| State                   | What's happening                     | You have data? |
| ----------------------- | ------------------------------------ | -------------- |
| `invalidated-immediate` | Data arrived but was already stale   | Yes            |
| `invalidated-initial`   | Marked stale, re-fetch hasn't started | Yes            |
| `invalidated-pending`   | Re-fetch is in progress              | Yes            |

Inside `<Await>`, you don't need to think about these states individually.
The second argument — `isInvalidated` — tells you everything:

```tsx
<Await store={store}>
  {(data, isInvalidated) => (
    <div style={{ opacity: isInvalidated ? 0.6 : 1 }}>
      <Dashboard data={data} />
    </div>
  )}
</Await>
```

Data stays on screen. Users see something useful instead of a spinner. One boolean to style it.

## Exhaustive matching

Most of the time you'll use `<Await>` and never touch these states directly.
But when you need custom control flow — logging, analytics, conditional rendering — the
full discriminated union is there, and TypeScript enforces exhaustive handling:

```ts
function describe<T>(rd: RemoteData<T>): string {
  switch (rd.type) {
    case 'initial':               return 'Not started';
    case 'pending':               return 'Loading...';
    case 'failed':                return `${rd.errors.length} error(s)`;
    case 'success':               return `Got: ${rd.value}`;
    case 'invalidated-immediate': return `Stale (just arrived): ${rd.invalidated.value}`;
    case 'invalidated-initial':   return `Stale (waiting): ${rd.invalidated.value}`;
    case 'invalidated-pending':   return `Refreshing: ${rd.invalidated.value}`;
    // No default — TypeScript errors if you miss a case.
  }
}
```

If you use [ts-pattern](https://github.com/gvergnaud/ts-pattern), it works as you'd expect:

```ts
import { match } from 'ts-pattern';

const label = match(store.current)
  .with({ type: 'initial' }, () => 'Idle')
  .with({ type: 'pending' }, () => 'Loading...')
  .with({ type: 'failed' },  ({ errors }) => `${errors.length} error(s)`)
  .with({ type: 'success' }, ({ value }) => `Hello, ${value.name}`)
  .with({ type: 'invalidated-immediate' }, ({ invalidated }) => `Stale: ${invalidated.value.name}`)
  .with({ type: 'invalidated-initial' },   ({ invalidated }) => `Stale: ${invalidated.value.name}`)
  .with({ type: 'invalidated-pending' },   ({ invalidated }) => `Refreshing: ${invalidated.value.name}`)
  .exhaustive();
```

## Prior art

The `RemoteData` pattern originates in [Elm](https://package.elm-lang.org/packages/krisajenkins/remotedata/latest/)
and has been adopted across typed functional languages.
The core idea — model async state as a sum type instead of a bag of booleans — is the same
principle behind [zod](https://zod.dev/) (parse, don't validate),
[neverthrow](https://github.com/supermacro/neverthrow) (errors as values),
and [ts-pattern](https://github.com/gvergnaud/ts-pattern) (exhaustive matching).
If you already think in discriminated unions, `RemoteData` is the natural way to model async data.
