# Typed Errors

:::info
This is an advanced feature. Most apps don't need it — the default error handling described
in [Failures and Retries](failures-retries) covers the common case. Read on if your API returns
structured, domain-level error objects that you want to handle differently from unexpected crashes.
:::

Some APIs — especially GraphQL — return **typed error objects** as part of their normal response.
A `fetchPerson` call might return a `Person`, a `PersonNotFound`, or a `PersonDeleted`.
These aren't crashes. They're expected outcomes with structured data.

By default, `use-remote-data` treats all failures the same: a rejected promise means something went wrong.
But with `useRemoteDataEither` (or `useRemoteDataMapEither` for [dynamic data](dynamic-data)),
you can separate **domain errors** (like "not found") from **unexpected errors** (like "network failed").

This gives you `RemoteDataStore<T, E>`, where `E` is your custom error type.

## How it works

Instead of returning a plain value, your fetch function returns an `Either` — a simple tagged object
that represents one of two outcomes:

- `Either.right(value)` — the happy path (success)
- `Either.left(error)` — a domain-level failure (e.g. "not found", "deleted")

Think of it like returning `{ tag: 'right', value: person }` or `{ tag: 'left', value: notFoundError }`.
The `Either.right()` and `Either.left()` helpers just create these objects for you.

When the store is in the `Failed` state, its `errors` array contains `Either<WeakError, E>` items:
- **Left** (`WeakError`) — unexpected errors like network failures or thrown exceptions. `WeakError` is just `Error | unknown`, because JavaScript lets you `throw` anything.
- **Right** (`E`) — your typed domain errors.

This lets your error component render each case differently.
